# Dev Docs Methodology

Status: Interpretive Note
Scope: `dev-docs/` as a non-normative interpretive envelope over existing repository law

These documents are interpretive and explanatory.

They do not introduce new primitives, alter authority order, or redefine canonical behavior.

All concepts written here must map to existing repository law.

## Purpose

The purpose of `dev-docs/` is to protect methodology without duplicating machinery.

This directory exists to:

- explain structural intuitions without elevating them to law
- preserve speculative and philosophical framing without creating a second system
- map abstract models back to implemented or constitutionally frozen repository structures
- reduce drift where projection, UI, protocol, or metaphor are mistaken for ontology

This directory does not exist to:

- define runtime authority
- mint new primitives
- rename existing law into parallel vocabulary
- replace `docs/` with a second normative layer

## Authority Order

Authority remains unchanged:

1. replayable deterministic runtime behavior
2. normative repository law under `docs/`
3. witness and generated artifact surfaces
4. interpretive material under `dev-docs/`

If `dev-docs/` and existing repository law diverge:

1. verify replayable behavior
2. defer to `docs/` and executable law
3. revise `dev-docs/`

`dev-docs/` has no upward authority.

## Boundary Rule

`dev-docs/` is an interpretive envelope over the system.

It is not:

- a source of new primitives
- a second ontology
- a replacement lexicon
- a parallel schema set
- a place to silently redefine identity through prose

Interpretation is allowed only when it remains downstream of existing law.

## Required Distinction

Every file in `dev-docs/` should maintain the distinction between:

- actualities: what is implemented, emitted, frozen, or already normative
- speculative possibilities: models, analogies, future possibilities, and interpretive frames

Speculative material must be marked as speculative and must not be written as if already implemented.

## Canonical Working Rule

When a `dev-docs/` note introduces an abstraction, it must answer:

1. What existing repository structure, surface, or rule does this describe?
2. Is this an actuality map, interpretive note, speculative model, or future possibility?
3. What does this abstraction not change?
4. Where is the normative source?

If those answers are not available, the abstraction is not ready to keep.

## Writing Template

New `dev-docs/` files should follow this shape:

1. Concept
2. Interpretation
3. Mapping to repo law
4. Non-goals / limits

Practical top labels:

- `Status: Actuality Map`
- `Status: Interpretive Note`
- `Status: Speculative Model`
- `Status: Future Possibility`

## Mapping Discipline

Abstract language is allowed only when it maps back to existing law.

Examples of acceptable material:

- structural models such as affine/projective or matroidal interpretation
- symbolic dialect explanations such as Braille, hexagram, or `x/X`
- anti-pattern notes that prevent UI, transport, or protocol drift
- evaluation strategy interpretations such as `LOW_LAW` and `HIGH_EDIT`
- future-facing models for convergence, reconciliation, or control-plane multiplexing

Every such note must end in a direct mapping section that points back to current repository law.

## Non-Duplication Rule

`dev-docs/` must not restate existing normative definitions as if they originate here.

In particular, `dev-docs/` should not independently redefine:

- claim
- proposal
- closure
- receipt
- provenance
- artifact identity
- canonical, golden, or negative surfaces
- primitive types already fixed by repository law

If a note needs those terms, it should cite and map to the existing normative documents instead.

## Anti-Drift Rule

`dev-docs/` should actively prevent predictable failure modes.

The main failure modes are:

- treating philosophy as a second system
- treating UI order as ontology
- treating transport or encoding as truth
- treating artifacts as identity-creating rather than identity-witnessing
- treating speculative framing as implemented law
- duplicating the same function under different words

## What Dev Docs Is Not

`dev-docs/` is not:

- a messaging protocol spec
- a browser-state authority layer
- a graph-first redefinition of the system
- a semantic-web-first replacement ontology
- a consensus theory of truth
- an excuse to move implementation drift into philosophical prose

## Design Invariants To Preserve

The interpretive layer should preserve these repository-level invariants:

- replay remains sovereign
- authority remains typed and deterministic
- artifacts witness truth; they do not create it
- projection is downstream of law
- identity is derived from lawful replay, not presentation order
- closure and reconciliation must remain law-derived rather than narrative-derived

These are preservation statements, not new definitions.

## Mapping To Repo Law

This methodology note derives from, and must stay subordinate to, existing repository law including:

- `AGENTS.md` for repository-wide authority order and replay sovereignty
- `docs/ONTOLOGY.md` for derived structure, type constraints, step-centric flow, and the affine/projective plus `LOW_LAW` / `HIGH_EDIT` distinctions
- `docs/PROPAGATION_CLAIMS_AND_FEDERATION.md` for claim/proposal/closure/receipt language, propagation, translation, provenance, reconciliation, and non-goals
- `docs/NARRATIVE_WITNESS_BINDING.md` for a direct example of an advisory overlay that remains downstream of canonical authority

## Non-Goals / Limits

This note does not:

- change runtime behavior
- create normative terminology
- settle open research questions
- grant `dev-docs/` any governance role over `docs/`
- authorize speculation to bypass replay or deterministic evidence

## Short Form

`dev-docs/` is a controlled space for interpretation.

Law stays singular.

Interpretation may be plural, but only as a mapped witness to existing repository law.

Status: Interpretive Note
Scope: how symbolic forms can be discussed without turning them into law

## Concept

The repository uses symbolic dialects such as Braille, hexagram language, and `x/X` class notation.

These dialects are meaningful.

They are not sovereign.

## Interpretation

A symbolic dialect is a witness vocabulary or projection vocabulary that helps express structure already derived elsewhere.

In this repository, symbolic dialects are best understood as:

- a disciplined way to encode or witness derived structure
- a way to preserve readable patterns across multiple surfaces
- a bridge between derivation and human interpretation

They are not:

- primary truth-makers
- substitutes for replay
- authority sources independent of typed deterministic derivation

### Braille

Braille is useful here as a witness encoding because it preserves bitwise distinctions in a compact symbolic form.

Its value is not decorative.

Its value is that it can act as a disciplined witness surface over already-derived structure.

### Hexagram and similar symbolic state forms

Hexagram language is useful when contributors need to think about state classes, transitions, or witness forms in a compact symbolic frame.

That usefulness is interpretive.

It does not create state law.

The `xx/xX/Xx/XX` family is the repository’s already-declared primary material class system for shared artifact and block surfaces.

In interpretive writing, it may help to describe these as mixed affine/projective material witnesses.

# Affine vs Projective

Status: Interpretive Note
Scope: explaining an existing surface distinction without redefining it

## Concept

`affine` and `projective` already exist in repository law as a surface distinction.

This file explains how to think about that distinction without treating it as a new ontology.

## Interpretation

The practical intuition is:

- affine names local replay-continuous state substance
- projective names shared agreement or carrier-facing coordination surfaces

This is useful because it helps prevent two common mistakes:

- treating shared coordination as if it were the local replay substance itself
- treating local replay substance as if it were already a shared carrier or agreement surface

The distinction is not about which side is “more real.”

It is about which kind of role a surface is playing.

### Why the distinction matters

If affine and projective are collapsed, contributors start confusing:

- local step-instantiated identity with shared coordination
- runtime continuity with publication or exchange
- state substance with witness or carrier material

That confusion usually leaks into UI, transport, or artifact naming.

### Why the distinction is still not enough by itself

Affine/projective is one axis only.

It should not be silently merged with evaluation-strategy language such as `LOW_LAW` and `HIGH_EDIT`.

Surface and evaluation remain different questions.

## Mapping To Repo Law

This interpretation maps directly to:

- `docs/ONTOLOGY.md`, section `5A. Surface Axis and Evaluation Axis`
- `docs/PROPAGATION_CLAIMS_AND_FEDERATION.md`, where affine/projective and `LOW_LAW` / `HIGH_EDIT` are explicitly kept separate
- `docs/SURFACES.md`, which keeps projection, transport, artifact, and witness layers distinct from canonical replay

The key mapping statements already frozen in repository law are:

- affine/projective is a surface distinction
- `LOW_LAW` / `HIGH_EDIT` is an evaluation-strategy distinction
- projective coordinates identity
- affine instantiates identity

## Non-Goals / Limits

This file does not:

- redefine affine or projective
- introduce extra affine/projective subtypes
- claim that the distinction alone explains the whole system

#+end_src

This file explains how to think about convergence and reconciliation in repository terms.

- convergence is where multiple downstream claims, receipts, or witnesses are brought into comparison

# UI As Projection

Status: Actuality Map
Scope: practical reminder that UI remains a downstream consumer

## Concept

This file restates a narrow actuality already present in repository law:

the UI is a projection consumer only.

## Interpretation

The browser, DOM, scene graph, frame picker, and media controls are all useful because they make downstream structure visible.

They are not authoritative because they are:

- presentation surfaces
- control adapters
- witness consumers
- reconciliation displays

This is not an anti-UI claim.

It is a boundary claim.

The repository needs UI surfaces.

It does not allow those surfaces to redefine step order, identity, schema, or runtime law.

### Common drift pattern

Contributors often start inferring ontology from:

- DOM ordering
- visible panels
- projection-local hashes
- selected frame state
- browser-safe numeric coercions

The UI boundary exists precisely to prevent those mistakes.

## Mapping To Repo Law

This file maps directly to:

- `docs/PROJECTION_UI.md` for the explicit rule that the UI is a projection consumer only
- `docs/NARRATIVE_WITNESS_BINDING.md` for the witness-page rule that projection controls may change presentation and visibility only
- `docs/SURFACES.md` for the category rule that projection surfaces expose derived structure and do not define canonical state

Practical current examples in repository law include:

- `data-ttc-*` metadata as derived projection metadata only
- `resolved_step_identity` and `ui_frame_resolution` as downstream projection objects
- SVG, canvas, A-Frame, and timed media as projection surfaces rather than runtime authorities

## Non-Goals / Limits

This file does not:

- redefine the UI contract
- introduce new UI primitives
- authorize browser state to influence runtime truth

# Anti-Patterns

Status: Interpretive Note
Scope: naming the most likely mistakes so contributors do not recreate a second system in prose

## Concept

This file lists failure modes that `dev-docs/` should actively resist.

## Interpretation

The main role of this file is defensive.

It identifies ways interpretive writing can drift into governance, ontology, or implementation distortion.

## What This System Is Not

This system is not:

- a messaging protocol
- a UI-driven ontology
- a graph-first replacement law
- a semantic-web-first truth system
- a consensus-based truth system
- a prose-defined alternative to deterministic replay

## Anti-Patterns

### Philosophy as second system

Bad move:

- writing speculative language as if it were a new primitive layer

Why it is bad:

- it duplicates law and encourages ontology drift

### Projection as authority

Bad move:

- treating UI order, DOM state, scene graphs, or frame selection as truth-making

Why it is bad:

- projection is downstream and may not define structure or canonical state

### Transport as ontology

Bad move:

- inferring semantics from a carrier, exchange form, or scannable variant

Why it is bad:

- translation changes carrier or witness surface without changing canonical identity

### Artifact as identity-maker

Bad move:

- assuming an artifact path, package, or witness encoding creates truth by itself

Why it is bad:

- identity is derived from lawful replay and declared artifact law, not from presentation convenience

### Parallel vocabulary drift

Bad move:

- renaming existing concepts under new words and then treating the rename as insight

Why it is bad:

- it creates duplication without increasing precision

### Speculation written as actuality

Bad move:

- writing future-facing models as if they are already implemented

Why it is bad:

- it collapses actualities and possibilities into one misleading layer

## Mapping To Repo Law

These anti-patterns map back to:

- `AGENTS.md` for replay sovereignty, authority order, and the rule that docs and projections do not outrank executable deterministic replay
- `docs/SURFACES.md` for the separation of substrate, runtime, witness, projection, transport, and artifact layers
- `docs/PROJECTION_UI.md` for the UI-downstream rule
- `docs/PROPAGATION_CLAIMS_AND_FEDERATION.md` for the non-goals around consensus, projection authority, and layer collapse
- `dev-docs/00-methodology.md` for the local rule that `dev-docs/` must explain without redefining

## Non-Goals / Limits

This file does not:

- add new governance rules
- replace existing non-goals in `docs/`
- authorize `dev-docs/` to police runtime behavior beyond interpretive clarity

# Narrative Role-to-Carrier Address Reconstruction

Status: Interpretive Note
Scope: companion reconstruction of the audited mapping using the repository’s address/material vocabulary

## First-Principles Address Reconstruction

1. **Typed deterministic replay is sovereign.** Every coordinate (address, material class, carrier witness) must originate from the replay trace, not from downstream prose. `[runtime-derived]` `[[docs/ONTOLOGY.md:3-56]]`
2. **Address/material witness (`xx/xX/Xx/XX`)** comes from `ttc_material_class_from_input_tick`. This is the canonical block/artifact coordinate generated from the carrier tuple, not an interpretive label. `[runtime-derived]` `[[src/ttc_carrier.c:16-20]]`
3. **State evaluation axis (`LOW_LAW/HIGH_EDIT/NULL_VOID`)** is derived from `ttc_state_class_from_state8`. It is part of the carrier coordinate set but is treated as a separate evaluation axis, not the primary address contract. `[runtime-derived]` `[[src/ttc_carrier.c:6-14]]`
4. **Carrier resolution (closure class, point_or_region, deterministic closure, resolved scope)** is computed next by `ttc_carrier_resolution_from_tuple`. This step never touches symbolic role. `[runtime-derived]` `[[src/ttc_carrier.c:43-102]]`
5. **Only after the tuple + resolution exist** may downstream policy overlay symbolic roles (`claim`, `proposal`, etc.). `[interpretive]`
6. **Boundary sentence:** A first-principles reconstruction of addressing must begin with replay-derived carrier coordinates and only then describe downstream role overlays; otherwise symbolic role is allowed to masquerade as address law. `[interpretive]`

## Runtime-Derived Coordinate Tuple

- `material_class` = `xx/xX/Xx/XX` is the canonical block/artifact witness derived from the input byte’s high bit and the tick parity. `[runtime-derived]` `[[src/ttc_carrier.c:16-20]]`
- `state_class` = `LOW_LAW`/`HIGH_EDIT` comes from the `state8` register bits (`0x0C`). `[runtime-derived]` `[[src/ttc_carrier.c:6-14]]`
- `carrier_resolution` (closure class, point_or_region, deterministic closure, closure scope) is a deterministic function over the tuple above, so it is the next coordinate in the chain before any narrative tag is applied. `[runtime-derived]` `[[src/ttc_carrier.c:43-102]]`
- These fields match the block/artifact hierarchy precisely: the runtime logic never references claim/proposal/closure/receipt, so the coordinate tuple stands alone as the minimal witness the binder must preserve. `[interpretive]`

## Where Narrative Assignment Begins

- `NARRATIVE_CARRIER_MAPPING` copies the runtime tuple into each narrative artifact and then appends `carrier_resolution`. `[downstream-binding]` `[[scripts/narrative/bind_narrative_to_witness.py:29-241]]`
- The mapping is applied after the runtime tuple is already known; the binder never reads carrier inputs to derive these fields—it simply overlays the known tuple onto the narrative step based on artifact class. `[downstream-binding]`
- Frame export reuses the same overlay to emit `closure` and `receipt` records, so the policy now affects both the narrative bundle and projection receipts. `[downstream-binding]` `[[scripts/narrative/export_narrative_frames.mjs:1-208]]`

## Address Preservation vs Address Forcing

- Preservation: `material_class`, `state_class`, and `carrier_resolution` remain untouched between runtime emission and narrative witness because the binder copies them wholesale. `[downstream-binding]`
- Forcing: `closure` is always labeled `xX`/`LOW_LAW` and `receipt` is always `Xx`/`LOW_LAW` even though runtime carriers could produce the other combinations of `xx/xX/Xx/XX`. The mapping therefore forces a specific address coordinate in the narrative export, reducing the tuple to a fixed assignment rather than a contextual witness. `[interpretive]`
- The exporter/validator path (the manifest and receipts) freezes the forced tuple as well, making the forensic pipeline assert the same address values multiple times. `[downstream-binding]` `[[scripts/narrative/validate_narrative_frame_export.py:45-210]]`

## Implications for the Existing Role Mapping

- The audit already established that no runtime function derives the role-to-tuple mapping, so this reconstruction restates the same conclusion in the address/material vocabulary: the coordinate tuple is primary, symbolic role is a later overlay. `[interpretive]`
- If we keep the tuple as a witness and describe narrative roles as overlays, we preserve the block/artifact hierarchy while still explaining why `claim` maps to `xx/LOW_LAW/point`, etc. `[interpretive]`
- The note shows exactly where the transition occurs: the binder adds roles after copying the tuple, the exporter repeats that overlay, and the validators enforce the overlay as deterministic downstream material. `[downstream-binding]`

## Short Verdict

The auxiliary conclusion is: the minimal runtime-derived coordinate tuple must stay sacrosanct (address/material witness + state + carrier resolution) and the narrative role mapping should be described as a downstream overlay on top of those coordinates, not as a new address definition. `[interpretive]`

# Narrative Role-to-Carrier Audit

Status: Interpretive Note
Scope: research-only audit of the narrative binders’ carrier convention

## Convention under review

The current pipeline maps each narrative artifact class (`claim`, `proposal`, `closure`, `receipt`) to a fixed tuple of `material_class`, `state_class`, and carrier fields such as `closure_class`, `point_or_region`, and `deterministic_closure`. This audit treats that convention as downstream binding law and distinguishes it from runtime truth. `[downstream-binding]`

## What runtime proves

- `ttc_material_class_from_input_tick` derives `xx`, `xX`, `Xx`, `XX` by combining the high bit of the input byte with the tick parity, so the runtime carrier surface class is purely a function of the raw event carrier, not of any narrative artifact role. `[runtime-derived]` `[[src/ttc_carrier.c:16-20]]`
- `ttc_state_class_from_state8` inspects the `state8` byte to emit exactly `LOW_LAW` or `HIGH_EDIT` (and `NULL_VOID` for zero); that bit mask is the only source of the evaluation axis seen by runtime. `[runtime-derived]` `[[src/ttc_carrier.c:6-14]]`
- `ttc_carrier_closure_class_name`, `ttc_carrier_point_or_region_name`, and `ttc_carrier_resolution_from_tuple` compute `closure_class` (`deterministic_point`, `deterministic_projection`, `open_region`, `candidate_region`), `point_or_region`, `closure_rank`, and `deterministic_closure` entirely from the runtime `state_class` and `material_class` produced above; nothing in those functions inspects narrative roles. `[runtime-derived]` `[[src/ttc_carrier.c:43-102]]`
- `resolved_scope`/`closure_rank`/`deterministic_closure` are set to deterministic constants derived from the combination of `material_class` and the parity/scope logic inside `resolved_scope_for_class` and the `LOW_LAW` flag inside `ttc_carrier_resolution_from_tuple`, so the runtime sets the numeric carrier fields without narrative input. `[runtime-derived]` `[[src/ttc_carrier.c:43-102]]`

## What narrative binding asserts

- `NARRATIVE_CARRIER_MAPPING` hardcodes the four tuple entries shown above for `claim`, `proposal`, `closure`, and `receipt`, including every field of `carrier_resolution`. There is no runtime hook connecting those artifact classes to the underlying carrier tuple other than this dictionary. `[downstream-binding]` `[[scripts/narrative/bind_narrative_to_witness.py:29-80]]`
- `narrative_carrier_witness` copies `material_class`, `state_class`, and the `carrier_resolution` map from the dictionary into every bound step. The binder never consults the event carrier or grammar to change those values. `[downstream-binding]` `[[scripts/narrative/bind_narrative_to_witness.py:235-241]]`
- The metadata emitted at the top of `demo/narrative/derived/narrative.bound.v0.ndjson` states that “narrative carrier witness is explicit downstream binding law, not replay authority,” documenting the convention as an advisory overlay. `[downstream-binding]` `[[demo/narrative/derived/narrative.bound.v0.ndjson:1]]`
- Every derived record includes the mapped tuple (`material_class`, `state_class`, `carrier_resolution`) so narrative consumers see the same values; the binder is the only place those numbers are introduced. `[downstream-binding]` `[[demo/narrative/derived/narrative.bound.v0.ndjson:3]]`

## Where validators/exporters enforce it

- `validate_narrative_binding.py` reruns the binder and insists the regenerated NDJSON/JS artifacts match the checked-in files byte-for-byte, so any change to the hardcoded mapping would be rejected by the reproducibility check. `[downstream-binding]` `[[scripts/narrative/validate_narrative_binding.py:158-179]]`
- The same validator compares the optional keys `material_class`, `state_class`, and `carrier_resolution` inside the live snapshot to themselves, raising an error if any of those witness values deviate between runs. That comparison makes the mapping part of the enforced witness determinism, not merely doctext. `[downstream-binding]` `[[scripts/narrative/validate_narrative_binding.py:191-208]]`
- `export_narrative_frames.mjs` independently reuses the same `NARRATIVE_CARRIER_MAPPING` to emit each frame manifest as a `closure` record and each projection receipt as a `receipt` record (including the same `material_class/state_class/carrier_resolution`). `[downstream-binding]` `[[scripts/narrative/export_narrative_frames.mjs:1-122]]`
- `validate_narrative_frame_export.py` runs the exporter twice, compares the manifests and receipts, and explicitly rejects any manifest or receipt whose `material_class`/`state_class` differs from `closure/xX/LOW_LAW` or `receipt/Xx/LOW_LAW`. `[downstream-binding]` `[[scripts/narrative/validate_narrative_frame_export.py:45-210]]`

- The same document emphasizes that affine/projective and `LOW_LAW`/`HIGH_EDIT` are orthogonal axes and that control-plane multiplex may separate artifact class, workflow scope, and field boundaries without creating truth. Hard-linking roles to classes effectively collapses those axes. `[interpretive]` `[[docs/PROPAGATION_CLAIMS_AND_FEDERATION.md:70-120]]`
- `docs/NARRATIVE_WITNESS_BINDING.md` restates “Typed deterministic replay is sovereign; narrative binding is an advisory witness overlay,” so any downstream mapping must not be mistaken for canonical behavior. `[interpretive]` `[[docs/NARRATIVE_WITNESS_BINDING.md:1-25]]`
- The binder metadata reiterates that authority stays deterministic and the narrative binding holds no replay mandate. `[interpretive]` `[[scripts/narrative/bind_narrative_to_witness.py:250-261]]`

- Plane A: 7-bit time over 8-bit carrier (current executable runtime).

Precedence policy:

1. Plane A runtime behavior is authoritative where executable code exists.

## 2) Plane A (Implemented): 7-bit Time / 8-bit Carrier

- Byte carrier uses framed trace bytes.
- Control bytes in assembler output:

# TTC Codebase Actual State v1 (March 27, 2026)

This document describes what the code **actually does now** in this repository, including drift points where behavior and expectations are not aligned.

## 1) Runtime Authority (What Is Authoritative)

Primary runtime authority is split across:

- `src/ttc_canonical_runtime.c`
- `scripts/ttc_busybox.sh`
- `scripts/materialize_trie_artifacts.sh`
- `scripts/leaf_contract.sh`

Projection/transport tools (non-kernel authority):

- `src/ttc_fano_aztec.c`
- `src/ttc_encode.c`
- `src/ttc_decode.c`
- `src/ttc_witness.c`

## 2) Addressing and Variant Classes (`xx/xX/Xx/XX`)

Current class computation is bitwise and deterministic in both main generators.

From busybox/materializer logic:

- `c0 = (input >> 5) & 1`
- `c1 = tick & 1`
- `class_idx = (c1 << 1) | c0`

Mapping:

- `0 -> xx`
- `1 -> xX`
- `2 -> Xx`
- `3 -> XX`

Path shape in current trie materializers:

- `artifacts/<class>/<point>/<lane>/<leaf>/`

`artifact_resolve.awk` packs these into one deterministic 23-bit address:

- `class(2) | point(2) | lane(6) | leaf(13)`

Important: resolver reads the **path**, not dotfile body content.

## 3) What `artifacts/` Is vs What `blocks/` Is (Current Behavior)

### `artifacts/`

Deterministic addressed instance tree intended for interpreter/runtime-facing consumption.

Each leaf is written with fixed surfaces:

- `.canon`
- `.block`
- `.artifact`
- `.bitboard`
- `.golden`
- `.negative`

### `blocks/`

Currently written as a mirror surface by `ttc_write_leaf_contract` (in `leaf_contract.sh`) under:

- `blocks/<class>/<point>/<lane>/<leaf>/`

This mirror is generated from the same computed tuple, not independently resolved.

## 4) Dotfile Participation in Resolve

Dotfiles are currently **not** used to derive identity/address.

- Resolve/import identity is path-derived (`artifact_resolve.awk`).
- Dotfiles are witness/declaration surfaces for that identity.

So today:

- Path decides identity.
- Dotfiles do not participate in address derivation.

## 5) Row Encoding in Dotfiles (Current)

`leaf_contract.sh` emits Braille rows.

### Artifacts leaf rows

Form:

- `FS ADDR GS LEFT US RIGHT RS`

(using Braille control tokens; no word labels).

### Blocks mirror rows

Form:

- `ROW_OPEN ADDR_OPEN ADDR ADDR_CLOSE PAYLOAD_OPEN LEFT US RIGHT PAYLOAD_CLOSE ROW_CLOSE`

All six dotfiles at one leaf share the same framing shape; `.negative` mutates payload bytes.

## 6) Matrix / Aztec Compatibility Functionality (Current)

There are two matrix-related paths, with `aztec` retained only as compatibility naming:

1. `ttc_canonical_runtime encode --matrix`
- Produces per-tick 27x27 ASCII matrix projections (`#`/`.`).
- `--aztec` is a compatibility alias only and does not mean standards Aztec.
- This is projection output, not leaf ABI/object writing.

2. `ttc_encode`/`ttc_decode`/`ttc_witness` and `ttc_fano_aztec`
- `ttc_encode -m slots` exports 60-slot symbols.
- `ttc_decode` imports 60-slot symbols back to bytes.
- `ttc_witness` emits witness semantics for later projection.
- `ttc_fano_aztec` accepts framed bytes (default frame size 16) and emits ascii/raw/json witnesses.

Important constraint:

- `ttc_fano_aztec` processes only complete frames.
- Short input smaller than frame size yields empty `steps`/empty placement.

## 7) What Is Currently Off (Drift List)

These are the main inconsistencies causing confusion.

1. README drift
- README still references old registry/meta/trace expectations that no longer match emitted leaves.

2. Legacy registry assumptions
- Some scripts still reference `blocks/registry/divisors_5040.tsv` fallback behavior.
- Runtime block lookup falls back to built-in defaults if registry is missing.

3. `.bitboard` content
- In `leaf_contract.sh`, `.bitboard` currently writes the same row structure as `.artifact` (not a distinct packed board witness payload).

4. `.canon` gating
- `.canon` is currently emitted every write, not gated to special milestones.

5. Legacy smoke checks
- `scripts/smoke_canonical_runtime.sh` still contains `meta.json` hashing expectations while primary leaf surfaces are dotfiles.

6. Artifacts vs blocks population can diverge operationally
- If users delete one tree manually, mirror assumptions may break until regeneration pipelines are rerun.

## 8) Minimal, Deterministic Regeneration Paths

### A) Runtime-board to artifact leaves

```bash
cat input.bin | ./bin/ttc_canonical_runtime encode --blocks blocks > /tmp/board.txt
./scripts/materialize_trie_artifacts.sh --board-file /tmp/board.txt --out-root artifacts --clear-targets
```

### B) Busybox direct leaf writes

```bash
printf '120 88 95\n' | ./scripts/ttc_busybox.sh --out-root artifacts
```

### C) Matrix compatibility position import/export boundary

```bash
cat artifact.bin | ./bin/ttc_encode -m slots > aztec.pos
cat aztec.pos | ./bin/ttc_decode > artifact.bin
cat aztec.pos | ./bin/ttc_witness -m ascii
```

## 9) Practical Interpretation for Meta-Interpreter ABI

Current code supports this split in practice:

- `artifacts/*/.artifact`: runtime-facing Braille capsule row
- `blocks/*`: shared framing/vocabulary mirror rows

But identity remains path-first today. If identity must be dotfile-first, resolver changes are still required.

## 10) Immediate Documentation Freeze Recommendation

Use this as the working truth until code/docs are reconciled:

1. Address identity is deterministic and path-derived.
2. Dotfiles are fixed six-surface declarations.
3. Artifacts rows are ABI-facing Braille capsules.
4. Blocks rows are framed mirror/reference capsules.
5. Aztec is projection/import-export at slot/board boundaries, not current leaf identity law.

# TTC Rule Framework v1 (Bytecode-Agnostic)

Status: normative extraction framework for archived conversational research.

## 1. Purpose

This framework normalizes archived research prose into deterministic rule IR consumable by a meta-circular interpreter without binding to Prolog/Datalog runtimes.

## 2. Precedence and Fail-Closed Policy

1. `src/` runtime behavior and `docs/ttc_canonical_spec_v1.md` remain authoritative for implemented semantics.
2. Rule packs in `research/rules/*.rules.ndjson` are normalized projections of archived research.
3. Any rule with `status != implemented` must not alter normative execution unless explicitly enabled by policy.
4. Unknown rule IDs, missing required fields, invalid status, malformed sources, or unknown rule-object keys are hard failures.

## 3. Rule IR (NDJSON)

Required keys for every rule object:

- `kind` (must be `ttc.rule.v1`)
- `rule_id` (stable identifier; e.g. `A15`, `A21.2`)
- `domain` (`alphabet|braille|projection|consensus|boot|complex`)
- `inputs` (array of symbolic inputs)
- `when` (array of predicates/guards)
- `then` (array of deterministic effects)
- `invariants` (array of must-hold assertions)
- `status` (`implemented|target_state|research_open|deprecated|research_open_inferred`)
- `sources` (array of `path:line`)

Optional inferred metadata keys:
- `inferred` (bool)
- `template_basis` (array of rule IDs)
- `inference_version` (string)

### Canonical ordering

Sort rule objects by `(domain, rule_id, source_order)` where:
- `source_order` is lexical order of `sources[0]` when present.
- Numeric rule IDs are compared numerically before suffixes (`A21` < `A21.2` < `A22`).

### Deterministic digest preimage

For each rule object, compute canonical preimage string in this field order:

`kind|rule_id|domain|inputs_join|when_join|then_join|invariants_join|status|sources_join`

Where all list joins use `\x1f` and all numeric values must be decimal strings.

## 4. Fact and Query Schemas

Fact object required keys:
- `kind` (must be `ttc.fact.v1`)
- `fact_id`
- `domain`
- `payload`
- `sources`

Query object required keys:
- `kind` (must be `ttc.query.v1`)
- `query_id`
- `select`
- `where`
- `order`
- `limit`

Policy object required keys:
- `kind` (must be `ttc.policy.v1`)
- `allow_status`
- `allow_inferred`
- `fail_on_unknown_rule`
- `fail_on_schema_error`

Example fact:

```json
{"kind":"ttc.fact.v1","fact_id":"runtime.opcodes","domain":"alphabet","payload":{"TICK_A":"0x01"},"sources":["src/ttc_vm.awk:39"]}
```

Example query:

```json
{"kind":"ttc.query.v1","query_id":"implemented_projection_rules","select":["rule_id"],"where":["domain=projection","status=implemented"],"order":["rule_id asc"],"limit":"256"}
```

## 5. Conflict Resolution

- If multiple archived occurrences map to the same A-number, preserve all as suffixed rule IDs (`A11.1`, `A11.2`, `A21.1`, `A21.2`).
- If archived rule semantics conflict with canonical runtime, set `status=research_open` or `target_state` and attach runtime reference in `invariants`.
- Non-explicit A-numbers (`A4-A9`, `A14`, `A28`) are template-inferred with `status=research_open_inferred`, `inferred=true`, and disabled by default policy.

## 6. Mapping Outputs

- Rule packs:
  - `research/rules/alphabet.rules.ndjson`
  - `research/rules/braille.rules.ndjson`
  - `research/rules/projection.rules.ndjson`
  - `research/rules/consensus.rules.ndjson`
  - `research/rules/boot.rules.ndjson`
  - `research/rules/complex.rules.ndjson`
- Traceability:
  - `research/rules/traceability.tsv`
- Runtime policy and example IO:
  - `research/rules/policy.ndjson`
  - `research/rules/facts.ndjson`
  - `research/rules/queries.ndjson`

## 7. Validation Requirements

1. Every explicit `A*` occurrence in archived sources must appear in `traceability.tsv`.
2. No duplicate `(source_doc:line, rule_id)` rows.
3. Every NDJSON line must include all required keys.
4. All `implemented` rules must cite at least one runtime/canonical anchor in `invariants` or `sources`.
